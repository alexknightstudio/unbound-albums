## Phase 1 ‚Äî Foundation

- **2026-07-15** ‚Äî Project git repo initialized at the project folder. It previously resolved to `/Users/alexknight`, which would have risked committing personal files to GitHub.
- **2026-07-15** ‚Äî GitHub repo `alexknightstudio/unbound-albums` is **private**; it will hold Supabase/Stripe config and is a commercial product.
- **2026-07-15** ‚Äî Scaffolded on Next.js 16 + Tailwind v4 (current versions as of build). Tailwind v4 configures via CSS `@theme` in `globals.css`, not `tailwind.config.ts` ‚Äî brand tokens live there.
- **2026-07-15** ‚Äî Supabase project region `us-west-2` (Oregon), Postgres 17.
- **2026-07-15** ‚Äî Database schema is managed as **migrations in the repo** (`supabase/migrations/`), never clicked together in the dashboard. Schema is code: reviewable, repeatable, rebuildable.
- **2026-07-15** ‚Äî Locked vocabularies (album status, size, order status, photo orientation) are Postgres **enums**; template codes and regen limits are **check constraints**. A bad value fails at write time instead of rendering a broken spread.
- **2026-07-15** ‚Äî Admin is a row in an `admins` table, not a JWT claim: revoking is a delete, and it cannot be forged client-side.
- **2026-07-15** ‚Äî `orders.album_id` uses `on delete restrict` (not cascade). A paid order must survive album deletion ‚Äî losing the record of something someone paid for is not an acceptable failure mode.
- **2026-07-15** ‚Äî Orders are **read-only to couples**. They are created by the Stripe webhook and advanced by the admin panel, both service-role paths. A client that can write an order is a client deciding what it paid.
- **2026-07-15** ‚Äî `share_slug` defaults to `gen_random_uuid()` hex (122 bits, CSPRNG) rather than a pgcrypto call ‚Äî unguessable with zero extension-schema dependency. Share-link *read access* is deferred to Phase 3, where it belongs.
- **2026-07-15** ‚Äî All four storage buckets (`originals`, `thumbs`, `renders`, `print`) are **private**. Wedding photos are never served from a public URL; the app issues short-lived signed URLs. `print` is admin-only.
- **2026-07-15** ‚Äî The 200-photo hard cap is enforced in the upload flow, not by a database trigger. A counting trigger races under parallel uploads; the failure mode (a 201st photo) does not justify the locking complexity. Revisit if it ever actually happens.
- **2026-07-15** ‚Äî RLS verified by adversarial test, not by trusting the migration: a second logged-in user could not read, target, steal, or plant albums, could not read `admins`, and could not forge an order; the rightful owner could still read and write their own. Test users deleted afterward.
- **2026-07-15** ‚Äî Next.js 16 renamed `middleware.ts` to `proxy.ts`. Session refresh + route protection live in `src/proxy.ts`. Every Supabase auth guide online still says middleware; they are out of date.
- **2026-07-15** ‚Äî Magic links verify a **token hash** server-side (`/auth/confirm`) rather than exchanging a PKCE code. The PKCE flow ties the link to the browser that requested it; couples share an inbox and open mail on a different device than they requested it from, so that failure mode is the normal case, not an edge case. Requires custom email templates.
- **2026-07-15** ‚Äî Supabase's built-in email sender is **unusable**: ~2 emails/hour on free tier, no custom templates, testing-only per their docs. Auth cannot run on it. **Resend** adopted now (BUILD_PLAN already picks it for Phase 5) via SMTP ‚Äî `smtp.resend.com:465`, username is the literal string `resend`, password is the API key.
- **2026-07-15** ‚Äî **Sender is temporarily `onboarding@resend.dev`**, Resend's test address, which ONLY delivers to the Resend account owner (alex@alexknightstudio.com). Every other recipient is silently dropped. Acceptable while Alex is the only tester. **Must become a verified domain before Phase 6 beta couples** ‚Äî this is a hard launch blocker, not a nicety.
- **2026-07-15** ‚Äî **Domain email deliberately NOT touched.** `unboundalbums.com` is in a state Namecheap's own docs don't describe: Private Email is "auto-configured" (DKIM, SPF, mail/autodiscover/autoconfig CNAMEs all present) but root MX points at their *email forwarding* servers (`eforward1-5.registrar-servers.com`), not the documented `mx1`/`mx2.privateemail.com`. **Mail to hello@unboundalbums.com arrives in Private Email webmail anyway** ‚Äî verified by Alex sending a real test. Mechanism unknown. Resend needs an MX on `send.send`, and Namecheap only permits subdomain MX when Mail Settings is switched to "Custom MX" ‚Äî which would drop the working (unexplained) MX set. Rearranging a working system nobody can explain was judged the larger risk. Revisit with Namecheap support before Phase 6.
- **2026-07-15** ‚Äî Vercel gets only the two `NEXT_PUBLIC_SUPABASE_*` vars. `RESEND_API_KEY` is not there: Supabase sends auth email server-side from its own SMTP config, so the app never needs it. Add it in Phase 5 when order emails are built.
- **2026-07-15** ‚Äî Auth config (site URL, redirect allow-list, SMTP, email templates) is managed in `supabase/config.toml` and applied with `supabase config push`, not clicked in the dashboard. Same reasoning as the schema: it's code.
- **2026-07-15** ‚Äî The album status machine is enforced by a **database trigger**, not just app code. Legal moves: `uploading‚Üíanalyzing‚Üígenerating‚Üíready‚Üíordered‚Üíshipped`, plus the single backward edge `ready‚Üígenerating` (the full-album regenerate). Verified that even the service-role key ‚Äî which bypasses RLS entirely ‚Äî cannot make an illegal jump.
- **2026-07-15** ‚Äî Status transitions are logged to `album_status_events` by the trigger, capturing `auth.uid()` as the actor. The app cannot forget to log, and the log is read-only to clients. `actor_id` is intentionally NOT a foreign key to `auth.users`: deleting an account must not erase the history of what happened.
- **2026-07-15** ‚Äî `src/lib/albums/status.ts` mirrors the DB transition table so the UI can reason about state without a round trip. **The database is the authority.** Nothing automatically catches drift between the two ‚Äî change both together.
- **2026-07-15** ‚Äî Pricing lives in `src/lib/albums/sizes.ts` as cents (integers, never floats). A test asserts the locked v1 prices, so changing one is a deliberate act rather than a typo. Size selection shows price at creation; actual payment is Phase 5.
- **2026-07-15** ‚Äî Vitest added for the layout engine and pricing/order math per CLAUDE.md. `npm test`.
- **2026-07-15** ‚Äî Uploads are resumable via Supabase TUS at 6MB chunks (their endpoint **requires** exactly 6MB for all but the last part ‚Äî not a tuning knob). 3 concurrent, 3 retries. The `photos` row is written only after the bytes land: a failed upload leaves an orphaned object (cheap), an orphaned row would be a broken photo in the album.
- **2026-07-15** ‚Äî Access tokens are refetched **per upload attempt**, not once per batch. 150 photos on a slow connection can outlast a token, and a stale one fails every remaining upload with an opaque 401.
- **2026-07-15** ‚Äî **`accept="image/jpeg,image/png,image/webp"` on the file input is load-bearing.** Since WebKit 0f7604 (2024), iOS Safari transcodes photo-library picks to JPEG *only* when accept explicitly restricts the type set. Widening it to `image/*` silently starts delivering raw HEIC that nothing downstream can decode. Adding `image/heic` inverts it again and makes Safari re-encode couples' JPEGs and PNGs *into* HEIC (WordPress 6.7 shipped this bug). A test asserts the attribute never mentions HEIC.
- **2026-07-15** ‚Äî **HEIC is not accepted in v1** (overriding BUILD_PLAN's "HEIC accepted"). Nothing in the stack can decode it: sharp's prebuilt binaries omit libheif over HEVC patent licensing; Supabase Edge Functions cap at 2s CPU; client-side WASM is ~5s/photo (12 minutes for 150, on a phone, in a tab iOS will kill). Supabase Storage transformations *can* decode HEIC but need Pro ‚Äî **and cap at 25MB/50MP, which Alex's own 17‚Äì31MB 60MP exports exceed**, so that path would not have worked anyway. HEIC that slips through (drag-and-drop ignores accept) is rejected with copy telling the couple to pick from their photo library. Untested claim: whether sharp can decode a *real* iPhone HEIC ‚Äî a synthetic one was inconclusive and no real HEIC was available.
- **2026-07-15** ‚Äî Thumbnails are generated **server-side** (`/api/photos/[id]/thumbnail`, sharp, 600px long edge). Not in the browser: iOS Safari caps canvas area at ~16.7MP, so a 60MP export would silently produce a blank thumbnail. Measured: 60MP/25MB ‚Üí thumbnail in ~310ms using 96MB RSS. The call is fire-and-forget ‚Äî a photo is safely stored regardless, and a thumbnail can be regenerated.
- **2026-07-15** ‚Äî Orientation is measured **after** `sharp.rotate()` applies EXIF, so it reflects what a viewer sees. Phone photos carry an orientation tag; Lightroom exports have it baked in with no tag. Both work.
- **2026-07-15** ‚Äî `SUPABASE_SERVICE_ROLE_KEY` is used only in `src/lib/supabase/admin.ts`, which imports `server-only` so the build fails if it's ever pulled into a Client Component. Needed because couples can read but not write the `thumbs` bucket.
- **2026-07-16** ‚Äî **Upload target changed from mobile-cellular to desktop-WiFi.** Alex's phone test uploaded 15 files with zero silent failures (15 files / 15 rows / 15 thumbnails, all consistent) and confirmed iOS transcodes HEIC‚ÜíJPEG on the way in. But throughput fell off a cliff around file 11 ‚Äî exactly where his camera roll's small phone photos (0.2‚Äì2.5MB) gave way to synced full-res exports (13‚Äì34MB). Cellular can't move 150 √ó ~17MB ‚âà 2.5GB in a reasonable time. Real couples curate their 150 from a photographer's gallery on a computer, so desktop + WiFi is both the realistic flow and the better one. No code change: the responsive web app is identical on desktop. Mobile remains the target for *viewing*/sharing (Phases 3, 4, 6). CLAUDE.md principle #4 ("test on iOS Safari") left intact ‚Äî still worth doing, just no longer the gating bar. BUILD_PLAN milestone revised.
- **2026-07-15** ‚Äî **Storage reality check, measured on real files:** Alex's exports median **17.5MB** (largest 31MB), 60MP. **150 photos ‚âà 2.5GB ‚Äî 2.5√ó the entire Supabase free tier (1GB).** CLAUDE.md's 2‚Äì4GB/album estimate was right. Pro ($25/mo, 100GB ‚âà 40 albums) is required before the real 150-photo milestone; testing with a handful stays free. Thumbnail generation also pulls each original back out of storage, so an album costs ~2.5GB egress against the free tier's 5GB/month.
- **2026-07-16** ‚Äî **Photo analysis reads the 600px thumbnails, not originals.** Composition, emotion, stage, and color judgment survive downscaling; a 60MP original doesn't fit the API's image limits anyway (and would cost ~100√ó the tokens). If a thumbnail is missing (fire-and-forget generation failed), the analyze route downscales the original in memory rather than skipping the photo.
- **2026-07-16** ‚Äî Analysis model is **claude-opus-4-8 with structured outputs** (JSON schema enforced by the API, not by hoping). Estimated ~$0.09 per 10-photo batch ‚Üí **~$1.40 per 150-photo album**, comfortably inside the <$6 target with room for layout + regens.
- **2026-07-16** ‚Äî **Analysis runs one batch (10 photos) per API-route invocation**; the browser loops until done. Serverless time limits never bite, progress ("Reading your photos. 84 of 150") falls out naturally, a failure loses one batch not the run, and a closed laptop resumes exactly where it stopped. The cost guardrail is structural: photos are selected with `analysis IS NULL` and written with the same guard ‚Äî a photo can never be analyzed twice.
- **2026-07-16** ‚Äî **`needs_correction` can never be true for a professionally edited photo**, even if the model contradicts itself and names a correction for one. Correcting an already-graded photo does damage; the derivation in `src/lib/ai/analysis.ts` enforces the precedence and a test pins it.
- **2026-07-16** ‚Äî Analysis prompt is versioned (`v1` in `src/lib/ai/prompts/analysis.ts`); every stored analysis records its prompt version + model, so a future prompt revision can re-target only photos analyzed under the old one. Alex's review corrections get committed as new prompt versions per BUILD_PLAN.
- **2026-07-16** ‚Äî **`ANTHROPIC_API_KEY` is required in `.env.local` and Vercel** and is NOT yet set (as of this entry). The pipeline is built and tested but cannot run against real photos until the key from console.anthropic.com is added.
- **2026-07-16** ‚Äî **The layout engine validates; the model proposes.** Claude returns a full spread plan (template codes + slot assignments + set-asides with reasons); `src/lib/engine/engine.ts` ‚Äî pure, no I/O ‚Äî enforces everything the DB can't: slots exactly filled, orientation fit (squares sit anywhere; a landscape never enters a portrait slot), no photo used twice, and **full accounting** (every photo either placed or set aside with a reason ‚Äî a photo that silently vanishes is a broken promise). Invalid plans are regenerated, never patched.
- **2026-07-16** ‚Äî Template slot contract lives in `src/lib/engine/templates.ts` as pure data ‚Äî 23 codes matching the DB check constraint (a test pins the match). Slot ids + orientation constraints defined now; Phase 3 gives each code geometry as a React component reading the same file.
- **2026-07-16** ‚Äî Spread budget: `photos √∑ 5`, clamped 3‚Äì30. 30 spreads is the priced base (150 photos ‚âà 5/spread); the 18-photo test album gets 4.
- **2026-07-16** ‚Äî **Generation persists nothing until a plan validates.** The generate route gives the model one honest attempt and one repair attempt (with the validator's full error list appended); a plan that fails twice returns "try again" with zero rows written. Existing spreads are replaced wholesale on success ‚Äî a half-written plan is not a plan.
- **2026-07-16** ‚Äî `set_aside_reason` is a column on photos (new migration), NOT inside the analysis jsonb ‚Äî analysis is the immutable vision output; set-aside is a layout decision that changes on regenerate. Albums also gained `regen_count` (0‚Äì1 check) for the one full-album regenerate guardrail; spreads already carried their 0‚Äì3.
- **2026-07-16** ‚Äî The analyze‚Üígenerate chain runs from the browser: analysis loop finishes, the same component fires generation, then refreshes into the plan view. An album stuck mid-flight (closed laptop) resumes automatically on page load from either status.
- **2026-07-16** ‚Äî The `ready` album page renders a raw plan review (spreads + template codes + slot thumbnails + the set-aside tray with reasons) ‚Äî the Phase 2 milestone surface for Alex's professional review. Phase 3's spread renderer replaces it.
- **2026-07-16** ‚Äî **Every thumbnail ever stored was corrupt, and nothing had noticed.** Uploading a raw Node Buffer through supabase-js inside a Next route coerces it to a UTF-8 string somewhere in the stack ‚Äî every byte ‚â• 0x80 becomes the replacement character, destroying the JPEG while keeping a plausible file size. Counted files and row counts all looked perfect; the first thing to ever *decode* a stored thumbnail was the analysis pipeline, which is how it surfaced. Fix: wrap uploads in a `Blob` (comment in the thumbnail route marks it load-bearing); all thumbnails regenerated from originals and verified by download-back-and-decode. **Lesson recorded: existence checks are not integrity checks ‚Äî verify blobs by reading them back.**
- **2026-07-16** ‚Äî **The layout call must stream.** A whole-album design run thinks for 2‚Äì3 minutes; the plain (non-streaming) request died with a connection error at ~3min. `messages.stream()` + `finalMessage()` holds the connection open; the same 2.5-minute call then succeeded. Analysis batches (~30s) stay non-streaming.

- **2026-07-16** ‚Äî **Album page model corrected (Alex):** an album is NOT n identical two-page spreads. - **2026-07-16** ‚Äî **Templates are geometry data, not components.** Each slot carries a rect in fractions of the spread canvas; ONE SpreadRenderer draws all 23 templates, and the print PDF will draw through the same component ‚Äî the WYSIWYG guarantee is structural, not aspirational.
